# ADR-001 — PXE Reboot Mechanism for Bare Metal Machines

**Status**: Proposed  
**Date**: 2026-06-26  
**Deciders**: @mgrzybek

---

## Context

Smeltry provisions bare metal machines via Tinkerbell. Once a machine is
provisioned and running its OS, it may need to be reprovisioned (OS reinstall,
node replacement, decommission). Tinkerbell requires the machine to boot in PXE
mode so it can receive a new workflow.

**Constraint**: the machines in the reference installation have **no BMC**
(no IPMI, iDRAC, iLO, Redfish). Out-of-band power cycling is therefore not
available. A software-level reboot trigger is required.

Tinkerbell exposes a gRPC API (`github.com/tinkerbell/tink`) that can signal
a machine but does not itself perform the reboot — the machine must cooperate.

---

## Decision Drivers

- No BMC available on target machines
- Must work from the Kubernetes management cluster (operator context)
- Must be compatible with machines running Flatcar or Ubuntu
- Minimal attack surface: the mechanism must not introduce a persistent
  privileged service that is exploitable outside of the operator's control
- A new repository (`smeltry-io/tinkerbell-reboot-agent`) will be created
  to host whichever approach is chosen

---

## Evaluated Approaches

### Approach A — Lightweight agent on each machine

A small daemon (Go binary or systemd unit) runs on every provisioned machine.
It watches the Tinkerbell gRPC API for a reboot signal targeting its own MAC
address / UUID, then calls `systemctl reboot` or `reboot -f`.

**Pros**
- No persistent SSH access needed; no credentials stored in Kubernetes
- The agent can be embedded in the Ignition/Butane config generated by machinecfg
- Works regardless of network topology (agent initiates outbound connections)
- Tinkerbell gRPC already provides the signalling channel; no new protocol needed

**Cons**
- A new binary must be built, tested, distributed, and kept up to date
- The agent must handle reconnects and back-off (Tinkerbell may restart)
- Requires the gRPC address of the Tinkerbell server to be known at provisioning
  time and injected into the Ignition config

**Prototype validation needed**: confirm that Tinkerbell's gRPC API can be used
to send a targeted signal to a single machine, and that the agent can subscribe
and react without consuming excessive resources.

---

### Approach B — SSH Job from the operator

`smeltry-operator` creates a Kubernetes Job that:
1. Reads the machine's IP from `status.serverIP` (or the CAPI `Machine` object)
2. Uses a private SSH key (stored in a `Secret`) to connect and run `sudo reboot`

**Pros**
- No persistent agent on machines; attack surface is zero when not reprovisioning
- Standard tooling; no new binary
- Can be implemented entirely inside `smeltry-operator` without a new repo

**Cons**
- Requires storing SSH private keys as Kubernetes Secrets (sensitive, rotation needed)
- SSH must be accessible from the management cluster network
- The machine must have the operator's public key pre-installed — machinecfg must
  be updated to inject it
- If the machine is already unhealthy (kernel panic, network issue), SSH may not
  be reachable

---

### Approach C — Tinkerbell gRPC API called directly from the operator

`smeltry-operator` calls the Tinkerbell gRPC API directly to mark the machine's
workflow as pending, relying on a Tinkerbell feature that signals the PXE client
on next boot. The machine must still be rebooted by another means.

**Assessment**: This approach does not solve the reboot problem — it only
prepares Tinkerbell for the next PXE boot. It must be combined with Approach A
or B. It is therefore not a standalone option.

---

## Recommendation

**Approach A (lightweight agent)** is the recommended direction, subject to
prototype validation.

Rationale:
- No persistent credentials in Kubernetes (avoids Secret management burden)
- Aligns with the Flatcar/Ignition model already used in machinecfg: the agent
  binary and its systemd unit are declared in the Butane config, built into the
  Ignition file, and installed at provisioning time — zero extra steps
- Once the agent is running, reprovisioning is fully automated: operator patches
  machine status in Netbox → machinecfg deletes Hardware object → operator
  creates new Job (machinecfg) → agent receives gRPC signal → reboots → PXE

Approach B remains a valid fallback if the Tinkerbell gRPC signal API turns out
to be insufficient or unstable.

---

## Prototype Validation Checklist

Before implementing the full Epic 3 / Epic 4 reboot integration:

- [ ] Confirm Tinkerbell tink-server gRPC API version used in the reference
      installation (`github.com/tinkerbell/tink` — check the tag)
- [ ] Identify the correct RPC call that signals "this hardware should reboot
      and await a new workflow" (likely `WorkflowService` or `HardwareService`)
- [ ] Implement a minimal Go agent that subscribes to the gRPC stream,
      filters by machine UUID, and calls `exec.Command("systemctl", "reboot")`
- [ ] Build the agent for `linux/amd64` and embed it as a Flatcar Ignition drop-in
- [ ] Deploy on a test machine; trigger a reboot from a script simulating the
      operator call; verify PXE boot occurs

---

## Consequences

- A new repository `smeltry-io/tinkerbell-reboot-agent` must be created after
  prototype validation (see ROADMAP.md Epic 9)
- machinecfg's Butane/Ignition templates must be updated to embed the agent
  binary and its systemd unit when `spec.os = flatcar`
- `smeltry-operator` will call the Tinkerbell gRPC API from the
  `ServerClaimReconciler` and `ClusterClaimReconciler` when a node replacement
  or reprovisioning is requested
- Approach B (SSH Job) is documented as a fallback and will be revisited if
  the prototype reveals blockers
